1. المالك يمكنه الموافقة على عقاره الخاص حرجة
app/Policies/PropertyPolicy.php:27, 47, 50

الوصف:

السياسة تسمح للمالك بتعليم عقاره كمباع أو مؤجر باستخدام markAsSold و markAsRented حتى لو العقار لم تتم الموافقة عليه بعد. هذا يتعارض مع منطق العمل.

منطق العمل المفقود:

  • العقار يجب أن يكون في حالة active ليتم تعليمه كمباع/مؤجر
  • لا يمكن للمالك بيع/تأجير عقار غير معتمد
  • يجب أن يمر العقار بعملية الموافقة أولاً
owner_id == user_id + No status check = CAN MARK AS SOLD

نتيجة الثغرة:

  • المالك يمكنه "بيع" عقار لم تتم الموافقة عليه بعد
  • تضليل المشترين المحتملين
  • تحويل الأموال بدون حق

💀 سيناريو الهجوم:

  1. مستخدم ينشئ عقار جديد (الحالة: pending)
  2. بمجرد إضافة العقار، يمكنه طلب الموافقة على عقاره
  3. أو الأسوأ: يمكنه مباشرةً تعليمه كمباع
  4. يحتال على مشترٍ محتمل

الحل:

// app/Policies/PropertyPolicy.php

public function markAsSold(User $user, Property $property): bool
{
    // التحقق من أن العقار في حالة active
    if ($property->status !== 'active') {
        return false;
    }
    
    return $user->is_admin || $property->owner_id === $user->id;
}

    }
    
    return $user->is_admin || $property->owner_id === $user->id;
}

3. المستخدم يمكنه طلب التواصل مع عقاره الخاص حرجة
app/Services/ContactService.php:11-33

الوصف:

لا يوجد تحقق من أن المستخدم الذي يرسل طلب التواصل ليس نفس مالك العقار. هذا يسمح للمالك بطلب التواصل مع نفسه!

user_id == property.owner_id = SELF CONTACT REQUEST

مشاكل هذه الثغرة:

  • إرسال إشعارات غير ضرورية للمالك
  • تلاعب بنظام طلبات التواصل
  • إمكانية استغلال رقم هاتف المالك

💀 سيناريو الهجوم:

  1. مستخدم يضيف عقار (مكتب عقار)
  2. بصفتك مستخدم آخر، تطلب التواصل مع العقار
  3. باستخدام ثغرة ثانية، يمكنك رؤية رقم هاتف المالك
  4. لأن صاحب العقار يمكنه "طلب التواصل مع نفسه"

الحل:

// app/Services/ContactService.php

public function create(array $data, int $userId): ContactRequestModel
{
    $property = Property::findOrFail($data['property_id']);
    
    // ✅ إضافة التحقق الجديد
    if ($property->owner_id === $userId) {
        throw new \Exception(
            'لا يمكنك إرسال طلب تواصل لعقارك الخاص'
        );
    }
    
    // التحقق من وجود طلب سابق
    $existingRequest = ContactRequestModel::where('property_id', $data['property_id'])
        ->where('user_id', $userId)
        ->whereIn('status', ['pending', 'approved'])
        ->first();

    if ($existingRequest) {
        throw new \Exception(
            'لديك طلب تواصل سابق لهذا العقار'
        );
    }

    return ContactRequestModel::create([
        'property_id' => $data['property_id'],
        'user_id' => $userId,
        'owner_id' => $property->owner_id,
        'name' => $data['name'],
        'phone' => $data['phone'],
        'message' => $data['message'] ?? null,
        'status' => 'pending',
    ]);
}

الوصف:

العملية تسمح بتعديل النفس؟ delete ❌ لا verifyUser ❌ لا banUser ❌ لا unbanUser ❌ لا toggleUserActive ❌ لا update ⚠️ نعم (خطأ)

💀 سيناريو الهجوم:

  1. أدمن يعدل دور نفسه من admin إلى user
  2. يفقد جميع صلاحياته
  3. لا يمكنه العودة (لا صلاحية)

// app/Policies/UserPolicy.php

public function update(User $user, User $targetUser): bool
{
    // الأدمن يمكنه تعديل أي مستخدم، لكن لا يمكنه تعديل نفسه
    return $user->isAdmin() && $user->id !== $targetUser->id;
}

// أو إذا كان التعديل الذاتي مسموح
public function update(User $user, User $targetUser): bool
{
    // المستخدم يمكنه تعديل نفسه، الأدمن يمكنه تعديل أي شخص
    return $user->isAdmin() || $user->id === $targetUser->id;
}
6. عدم التحقق من ملكية طلب التواصل عالية
app/Policies/ContactRequestPolicy.php:20-25

الوصف:

سياسة view تتحقق من أن المستخدم هو صاحب الطلب أو المالك، لكن:

  • لا تتحقق من حالة الطلب (pending/approved/rejected)
  • لا تتحقق من حالة العقار
  • لا تتحقق من تاريخ الطلب

الحل:

public function view(User $user, ContactRequest $contactRequest): bool
{
    // الأدمن يمكنه رؤية كل الطلبات
    if ($user->isAdmin()) {
        return true;
    }
    
    // المستخدم العادي
    $isOwner = $contactRequest->user_id === $user->id;
    $isPropertyOwner = $contactRequest->owner_id === $user->id;
    
    // فقط المالك يرى الطلبات المقبولة
    // المستخدم يرى طلباته فقط
    if ($isOwner) {
        return true;
    }
    
    if ($isPropertyOwner && $contactRequest->status === 'approved') {
        return true;
    }
    
    return false;
}

8. لا توجد قواعد انتقالات الحالة متوسطة
app/Services/PropertyService.php:188-219

الوصف:

لا يوجد تحقق من حالة العقار قبل تغييرها. يمكن:

  • تغيير عقار مرفوض إلى مؤجر مباشرة
  • الموافقة على عقار他已经 sold
  • رفض عقار already active
من حالة إلى حالة صحيح؟
pendingactive✅ صحيح
pendingrejected✅ صحيح
activesold✅ صحيح
activerented✅ صحيح
rejectedsold❌ خطأ
soldactive❌ خطأ
rentedsold⚠️ спорно

الحل:

private function canTransition(string $from, string $to): bool
{
    $rules = [
        'pending' => ['active', 'rejected'],
        'active' => ['rented', 'sold'],
        'rented' => ['sold', 'active'], // إذا ended rental
        'sold' => [], // حالة نهائية
        'rejected' => ['pending'], // يمكن إعادة التقديم
    ];
    
    return in_array($to, $rules[$from] ?? []);
}

public function approve(Property $property, int $adminId): Property
{
    if (!$this->canTransition($property->status, 'active')) {
        throw new \InvalidArgumentException(
            'لا يمكن الموافقة على عقار في حالة: ' . $property->status
        );
    }
    
    $property->update([
        'status' => 'active',
        'approved_by' => $adminId,
        'approved_at' => now(),
    ]);
    
    return $property;
}

public function approveContactRequest(Request $request, $id)
{
    $contactRequest = ContactRequest::find($id);
    
    if (!$contactRequest) {
        return $this->errorResponse('طلب التواصل غير موجود', 404);
    }

    if (!Gate::allows('approve', $contactRequest)) {
        return $this->errorResponse('غير مصرح', 403);

    // ✅ تحقق دقيق
    if ($contactRequest->status !== 'pending') {

        ];
        return $this->errorResponse(
            $messages[$contactRequest->status] ?? 'تم معالجة هذا الطلب',
            400
